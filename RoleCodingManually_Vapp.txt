🏗️ 1. معماری پروژه (Architecture)

ساختار پوشه‌ها:
- Controller/ - کنترلرهای API (ارث‌بری از VappControllerBase)
- Services/ - سرویس‌ها (Business Logic)
- Repositories/ - ریپازیتوری‌ها (Data Access)
- Models/ - مدل‌های دیتابیس (Entities)
- DTOs/ - Data Transfer Objects (شامل Common/ApiResponse، ErrorCodes)
- Exceptions/ - AppException
- Middleware/ - GlobalExceptionHandlerMiddleware
- Filters/ - ApiTraceIdResultFilter
- Utilities/ - ControlledErrorHelper، ErrorTranslator
- Data/ - DbContext و تنظیمات دیتابیس

اصول:
- استفاده از Dependency Injection
- Controller ها فقط Orchestration انجام دهند
- منطق کسب‌وکار در Service باشد

✅ 4. اعتبارسنجی‌ها (Validation)

Data Annotations در DTO:
- همیشه از پیام‌های خطای فارسی استفاده کنید:
```csharp
[Required(ErrorMessage = "نام الزامی است")]
[MaxLength(100, ErrorMessage = "نام نمی‌تواند بیشتر از 100 کاراکتر باشد")]
```

در Controller (استاندارد پروژه):

- Controllerها از VappControllerBase ارث‌بری کنند
- برای ModelState از ErrorTranslator استفاده کنید (هرگز ExceptionMessage خام نخوانید)
- errorCode اعتبارسنجی: VALIDATION_FAILED

```csharp
// ✅ درست
if (!ModelState.IsValid)
{
    var errors = ExtractModelStateErrors(); // از VappControllerBase
    return StatusCode(400, ApiResponse<T>.BadRequest(
        "داده‌های ورودی نامعتبر است",
        errors,
        ErrorCodes.ValidationFailed));
}

// ❌ غلط — هرگز
errorMessage = error.Exception.Message;
var errors = ModelState.Values.SelectMany(...).Select(e => e.ErrorMessage);
```

در Service:

- قوانین کسب‌وکار: throw AppException یا return ApiResponse با پیام ثابت فارسی
- هرگز ex.Message را به کاربر نفرستید

```csharp
// ✅ درست — خطای کنترل‌شده
if (amount <= 0)
    throw AppException.BadRequest(ErrorCodes.InvalidInput, "مبلغ باید بیشتر از صفر باشد");

// ✅ درست — خطای غیرمنتظره
catch (Exception ex)
{
    _logger.LogError(ex, "خطا در عملیات — {EntityId}", entityId);
    return ApiResponse<T>.InternalServerError(ControlledErrorHelper.Unexpected);
}

// ❌ غلط — هرگز
return ApiResponse<T>.InternalServerError($"خطا: {ex.Message}");
throw new Exception(ex.Message);
```

🚨 4.1. مدیریت خطا — قانون طلایی پروژه

اصل اصلی:
  خطای واقعی (دیتابیس، Exception، StackTrace) → فقط در لاگ سرور
  پیام کنترل‌شده (فارسی، ثابت)              → فقط برای کاربر

فایل‌های مرجع (قبل از نوشتن کد خطا، این‌ها را ببینید):
- Utilities/ControlledErrorHelper.cs  → پیام‌های امن کاربر
- DTOs/Common/ErrorCodes.cs           → کدهای خطای استاندارد
- Exceptions/AppException.cs          → Exception کنترل‌شده
- Middleware/GlobalExceptionHandlerMiddleware.cs
- Filters/ApiTraceIdResultFilter.cs
- Controller/VappControllerBase.cs

ساختار ApiResponse (الزامی):
```csharp
public class ApiResponse<T>
{
    public int StatusCode { get; set; }
    public bool Success { get; set; }
    public string Message { get; set; }
    public string? ErrorCode { get; set; }   // برای فرانت
    public T? Data { get; set; }
    public List<string>? Errors { get; set; }
    public string? TraceId { get; set; }   // برای پشتیبانی
}
```

ErrorCodes — کدهای استاندارد (از ErrorCodes.cs استفاده کنید، رشته آزاد ننویسید):
- UNEXPECTED_ERROR      → خطای غیرمنتظره سرور
- VALIDATION_FAILED     → خطای اعتبارسنجی فرم
- INVALID_INPUT         → ورودی نامعتبر
- INVALID_USER_ID       → شناسه کاربر نامعتبر
- UNAUTHORIZED          → دسترسی غیرمجاز
- FORBIDDEN             → ممنوع
- NOT_FOUND             → یافت نشد
- DATABASE_ERROR        → خطای ذخیره‌سازی
- TOKEN_EXPIRED         → توکن منقضی
- TOKEN_INVALID         → توکن نامعتبر
- SMS_FAILED            → خطای پیامک
- PAYMENT_FAILED        → خطای پرداخت
- FILE_UPLOAD_FAILED    → خطای آپلود

AppException — چه زمانی استفاده کنیم:
```csharp
// در Controller یا Service — خطای قابل پیش‌بینی
throw AppException.NotFound(ErrorCodes.NotFound, ControlledErrorHelper.NotFound);
throw AppException.Unauthorized(ErrorCodes.InvalidUserId, ControlledErrorHelper.Unauthorized);
throw AppException.BadRequest(ErrorCodes.InvalidInput, "نام الزامی است");
throw AppException.Internal(ErrorCodes.Unexpected, ControlledErrorHelper.Unexpected);
```
→ Middleware خودکار تبدیل به ApiResponse + traceId می‌کند

ControlledErrorHelper — پیام‌های امن (از این‌ها کپی نکنید، import کنید):
- Unexpected          → "خطای غیرمنتظره. لطفاً با پشتیبانی تماس بگیرید."
- Database            → "مشکلی در ذخیره‌سازی اطلاعات پیش آمد..."
- Unauthorized        → "شما مجاز به انجام این عملیات نیستید"
- NotFound            → "منبع مورد نظر یافت نشد"
- SmsFailed           → "مشکلی در ارسال پیامک پیش آمد..."
- FileUploadFailed    → "مشکلی در آپلود فایل پیش آمد..."

Controller — قوانین:
1. ارث‌بری از VappControllerBase (نه ControllerBase مستقیم)
2. GetCurrentUserIdAsync() از Base — خودش AppException می‌زند
3. try/catch در Controller ننویسید — Middleware handle می‌کند
4. همه endpointها ApiResponse<T> برگردانند (به جز موارد خاص تأییدشده)
5. ModelState: ExtractModelStateErrors() از Base

```csharp
public class ProductController : VappControllerBase
{
    public ProductController(IConfiguration config, IUserRepository repo, IProductService svc)
        : base(config, repo) { ... }

    [HttpPost]
    public async Task<ActionResult<ApiResponse<ProductDto>>> Create([FromBody] CreateProductDto dto)
    {
        if (!ModelState.IsValid)
        {
            var errors = ExtractModelStateErrors();
            return StatusCode(400, ApiResponse<ProductDto>.BadRequest(
                "داده‌های ورودی نامعتبر است", errors, ErrorCodes.ValidationFailed));
        }

        var userId = await GetCurrentUserIdAsync();
        var result = await _service.CreateAsync(userId, dto);
        return StatusCode(result.StatusCode, result);
    }
}
```

Service — قوانین:
1. خطای قابل پیش‌بینی → return ApiResponse با پیام ثابت یا throw AppException
2. catch (Exception ex) → فقط _logger.LogError(ex, ...) + پیام ثابت ControlledErrorHelper
3. هرگز: ex.Message، InnerException.Message، response.Message API خارجی را مستقیم به کاربر ندهید
4. پیام‌های API خارجی (SMS، پرداخت) → فقط در لاگ؛ به کاربر SmsFailed / PaymentFailed

Middleware (خودکار — دست نزنید مگر بدانید چرا):
- GlobalExceptionHandlerMiddleware → همه Exceptionهای handle نشده
- ApiTraceIdResultFilter → traceId روی همه ApiResponseها

فرانت‌اند (اگر مربوط باشد):
- فقط response.message بک‌اند را نشان بده
- response.errorCode برای منطق UI (مثلاً TOKEN_EXPIRED → logout)
- response.traceId را در گزارش خطا به پشتیبانی بده
- فقط وقتی API در دسترس نبود: "ارتباط برقرار نشد"

📦 18. مدیریت فایل

public async Task<string> UploadFileAsync(IFormFile file, string folder)
{
    var fileName = Guid.NewGuid() + Path.GetExtension(file.FileName);
    var filePath = Path.Combine("wwwroot/uploads", folder, fileName);
    
    using var stream = new FileStream(filePath, FileMode.Create);

✅ 19. پیام‌های نمایشی به کاربر

قانون: کاربر فقط پیام‌های کنترل‌شده فارسی می‌بیند — هرگز جزئیات فنی.

منابع مجاز پیام:
- ControlledErrorHelper.* (خطاهای عمومی)
- AppException با پیام فارسی عمدی (validation کسب‌وکار)
- Data Annotations در DTO (اعتبارسنجی فرم)

هرگز به کاربر نشان ندهید:
- ex.Message / exception.Message
- SqlException / StackTrace / InnerException
- مسیر فایل سرور (C:\... یا /var/...)
- پیام خام API خارجی (SMS، درگاه پرداخت)

موفقیت:
- "عملیات با موفقیت انجام شد"
- "اطلاعات با موفقیت ذخیره شد"

خطای عمومی (از ControlledErrorHelper):
- "خطای غیرمنتظره. لطفاً با پشتیبانی تماس بگیرید."
- "مشکلی در ذخیره‌سازی اطلاعات پیش آمد. لطفاً دوباره تلاش کنید."
- "شما مجاز به انجام این عملیات نیستید"
- "منبع مورد نظر یافت نشد"

اعتبارسنجی:
- "داده‌های ورودی نامعتبر است" + لیست errors
- "فیلد {نام فیلد} الزامی است"
- "مقدار وارد شده نامعتبر است"

پشتیبانی:
- همیشه traceId را در لاگ نگه دارید تا کاربر بتواند گزارش دهد
